# SmartStyle 👕

SmartStyle is a Flutter wardrobe assistant: users photograph their clothes, and
the app manages the wardrobe, generates **weather-aware weekly outfits with
Gemini**, analyzes wardrobe gaps, and plans day-by-day travel outfits — backed by
Firebase (Auth, Firestore, Storage) and Cloud Functions.

> **Why this README is detailed:** the interesting part of this project isn't
> "it calls an AI API" — it's the **prompt-engineered, schema-validated,
> wardrobe-grounded** outfit pipeline with a **full rule-based fallback** when
> the model is unavailable. The sections below show exactly how that works.

## Features

- **Auth** — Firebase Auth + Google Sign-In.
- **Wardrobe management** — photograph items; background is removed server-side
  (PhotoRoom) and color is analyzed; items carry category, type, color, fabric,
  season, style tags, and usage counts.
- **AI outfit suggestions** — weekly, weather-aware outfits generated by Gemini
  through a Cloud Function, grounded to the user's actual wardrobe (see
  [AI outfit pipeline](#ai-outfit-pipeline)).
- **Rule-based fallback** — if Gemini is slow, rate-limited, or returns invalid
  JSON, a deterministic local engine produces a valid outfit plan instead.
- **Wardrobe gap analysis** — computes *meaningful* combination counts (not raw
  multiplication) and suggests purchases that unlock the most new outfits.
- **Travel mode** — day-by-day packing and outfit planning per destination.
- **Favorites / saved outfits**, weather integration, and local notifications.

## Architecture

One-directional data flow. The Flutter UI never talks to Firestore or the AI
directly — it goes through a service, and AI runs inside Cloud Functions so the
API keys never ship in the app binary.

```
┌──────────────┐    ┌───────────────┐    ┌─────────────────────┐    ┌──────────────┐
│  Widgets     │ ─▶ │  Services      │ ─▶ │  Cloud Functions    │ ─▶ │  Gemini API  │
│ (screen/)    │    │ (services/)    │    │  (functions/, TS)   │    │  (server)    │
│              │ ◀─ │                │ ◀─ │                     │ ◀─ │              │
└──────────────┘    └───────┬────────┘    └──────────┬──────────┘    └──────────────┘
                            │                        │
                            ▼                        ▼
                    ┌───────────────┐        ┌─────────────────┐
                    │  Firestore /  │        │  PhotoRoom API  │
                    │  Storage      │        │  (bg removal)   │
                    └───────────────┘        └─────────────────┘
```

- **Widgets → Services:** screens read/write only through service classes
  (`WardrobeItemService`, `OutfitAiService`, `WeatherService`, `TravelModeService`, …).
- **Services → Firestore/Storage:** wardrobe items and outfits are stored per
  user in Firestore; raw photos land in Storage.
- **Services → Cloud Functions:** AI features are HTTPS-callable functions.
  `OutfitAiService` POSTs the wardrobe + weather + plan to
  `generateOutfitSuggestions`; keys stay server-side.
- **Storage trigger:** uploading a raw photo fires `onObjectFinalized`, which
  calls PhotoRoom to remove the background and writes back a processed PNG +
  status to Firestore. Item color is analyzed on-device (ML Kit) and via the
  `analyzeClothingColor` function.

### Why a Service layer but no client-side Repository?

Services already isolate all I/O (Firestore, Storage, HTTP-to-Functions) behind
plain injectable classes — every screen depends on a service, not on Firebase
types directly. Adding a Repository on the client would only wrap those services
without changing the source of truth. The real "domain logic" boundary in this
app is the **Cloud Functions layer** (validation, prompting, fallback), which is
where a heavier orchestration/repository pattern actually lives — in TypeScript,
server-side, testable, and outside the mobile bundle.

## AI outfit pipeline

**Cloud Function:** `functions/src/generateOutfitSuggestions.ts`
**Model:** `gemini-2.5-flash-lite` (Google Generative Language API)

**1. What is sent to Gemini** — *structured text, not photos.* The app sends a
JSON payload: the user's wardrobe (each item as `id`, `category`, `name`,
`color`, `season`, `style`, `fabric`), the requested days + events, the weather
per day, `userProfile.gender`, and `stylePriority`. Sending compact text (not
images) keeps latency and token cost low and makes grounding exact.

**2. Prompt engineering** — a long, rule-encoded **system prompt** defines the
stylist's behavior:
- **Grounding / anti-hallucination:** *"never suggest an item not in the
  wardrobe; use only the given `id` and `name`."*
- **Weather rules:** explicit temperature bands (e.g. ≥20 °C ⇒ outerwear must be
  `null`; ≤6 °C ⇒ heavy coat required).
- **Repetition rules:** per-category weekly reuse limits; no two consecutive
  identical top+bottom pairs.
- **Color-harmony hierarchy**, gender-specific formulas (e.g. weighted
  55/30/10/5 distribution for women's special events), one-piece dress rules,
  and a strict **output JSON schema**.
- Model config: `responseMimeType: application/json`, `temperature: 0.55`,
  `topP: 0.9`, `maxOutputTokens: 4096`.

**3. Response handling** — the model output is defensively processed:
`parseAndRepairJson` (strips code fences, extracts/repairs JSON) →
`validateGeminiOutput` (checks the schema, **drops any item id not in the
wardrobe**, filters gender-incompatible items) → `applyLocalPriorityRules`
(deterministic business rules applied *after* the model, e.g. shoe/priority
formulas for special and sport events).

**4. Fallback (rule-based engine)** — if `callGemini` throws (30 s timeout, HTTP
`429` quota, or unparseable JSON), the function calls
`buildLocalOutfitSuggestions`: a deterministic selector engine
(`selectTop`/`selectBottom`/`selectShoes`/`selectOuterwear`/`selectAccessory`)
that applies the same weather thresholds, gender/event priorities, and weekly
repetition tracking using Turkish/English keyword matching. The user always gets
a valid plan; the client (`OutfitAiService`, 45 s timeout) just renders it.

**Other AI functions:** `generatePurchaseSuggestions` (gap-filling purchases),
`generateWardrobeInsight` (wardrobe summary), `analyzeClothingColor`.

## The `recommendation_engine/` folder — what it is

This is a **standalone prototype scaffold** (not wired into the Flutter app) for
the "What Should I Buy?" purchase-recommendation feature — a Node/TS backend
(`intentParser`, `combinationValidity`, `fallbackEngine`, `geminiClient`,
`recommendPurchase`) plus a React Native flow. It's where that feature's
Gemini-plus-fallback logic was explored in isolation (it even has its own unit
test, `tests/combinationValidity.test.ts`).

The **production** purchase logic that the app actually calls lives in
`functions/src/generatePurchaseSuggestions.ts` (via `purchase_ai_service.dart`).
`recommendation_engine/` is kept as a reference/prototype and is intentionally
separate from the deployed backend.

## Security

- **Firestore rules** (`firestore.rules`) — every collection is **owner-scoped**:
  a signed-in user can only read/write documents whose `user_id` (or `userId`)
  equals their `auth.uid`. This covers the wardrobe collections
  (`ust_giyim`, `alt_giyim`, `dis_giyim`, `ayakkabi`, `corap`), saved outfits
  (`kombinler`, `users/{uid}/favorite_outfits`), and `travel_plans` including the
  `day_plans` subcollection (which verifies the parent plan's owner via
  `get()`/`getAfter()`).
- **API keys never ship in the client.** `GEMINI_API_KEY` and
  `PHOTOROOM_API_KEY` are declared as Cloud Functions secrets
  (`defineSecret(...)`) and injected at runtime — the Flutter app only knows the
  function URL, never the keys. `.env` / service-account files are git-ignored.

### Known gaps / next hardening (honest)

- **Function-call protection / App Check.** The AI functions are public HTTPS
  endpoints (`onRequest`, `cors: true`) and currently do **not** verify the
  caller — there is no Firebase **App Check** and no auth-token/`request.auth`
  check in the handlers. In practice anyone who discovers a function URL could
  POST to it and consume the project's Gemini / Firebase quota. The fix is to
  enforce App Check (and/or migrate these to `onCall` callable functions, which
  provide `request.auth` and built-in App Check enforcement) plus per-user rate
  limiting. *(The functions do validate and ground their input, but that
  protects output quality, not billing.)*
- **Storage security rules** are **not yet in the repo** (`firebase.json` deploys
  only `firestore.rules`). Since raw user photos land in Storage, owner-scoped
  Storage rules are a needed hardening step.

## Testing

```bash
flutter test          # Dart unit + widget tests
cd functions && npm test   # (if configured) Cloud Function tests
```

Current tests:
- `test/combination_logic_test.dart` — pure wardrobe logic (variant keys, color
  normalization, logical-combination counting). Good target because it's
  deterministic and framework-free.
- `test/travel_mode_service_test.dart`, `test/widget_test.dart`.
- `recommendation_engine/backend/tests/combinationValidity.test.ts` — prototype
  combination-validity rules.

CI (`.github/workflows/ci.yml`) runs `flutter analyze` + `flutter test`, and
type-checks (builds) the Cloud Functions on every push. Analyzer *errors* fail
the build; pre-existing *infos* (e.g. `withOpacity` deprecations) are a tracked
cleanup item.

## Getting started

```bash
flutter pub get
flutter run          # needs the Firebase config for project smartstyle-app-63929

# Cloud Functions
cd functions && npm install && npm run build

# secrets (server-side, never committed)
firebase functions:secrets:set GEMINI_API_KEY
firebase functions:secrets:set PHOTOROOM_API_KEY
```

## Project structure

```text
lib/
  core/theme/   App theming
  models/       Data models (WardrobeItem, outfit AI models, …)
  screen/       Screens (wardrobe, outfits, gap analysis, travel, auth)
  services/     Firestore/Storage/HTTP-to-Functions, AI, weather, wardrobe logic
  widgets/      Reusable UI
functions/
  src/          Deployed Cloud Functions (Gemini outfit/purchase/insight, color,
                Storage image pipeline)
recommendation_engine/
  backend/      Standalone "What Should I Buy?" prototype (Node/TS)
  frontend/     Prototype RN flow
firestore.rules Owner-scoped Firestore security rules
```
